GulfTIC CRM BRD

Foundational Data Model & Identity

The CRM is built on a "Conformed Data Model". The CR Number is the unique identifier ensuring data integrity.

Core Entity

🏢 Facility

The manufacturer/plant receiving the certificate. Created automatically when a Lead reaches "Qualified".

Rule: CR Number MUST be unique. Linked to one or many Deals.
Payer Entity

🤝 Client / Broker

The payer/communicator responsible for the transaction. Can be linked to multiple Facilities.

Rule: Mandatory "Is Broker?" (Yes/No) toggle required on creation.

The 360-Degree Unified View

Broker Profile

Displays a comprehensive list of all Facilities they manage.

Facility Profile

Displays a "Related Requests" table showing all active/closed deals across all brokers for that CR Number.

Deal Screen

Shows both the Facility and the Client/Broker side-by-side.

⚠ Data Integrity & Merge Tool

Restriction: Sales Reps DO NOT have the ability to scan the database or merge facilities.

A dedicated tool for Managers/Admins scans for matching CR Numbers or Fuzzy Names. Actions include Save New or Merge Old (Keep old, Merge contacts, companies, activities).

Detailed User Story Catalog & UAT

The technical requirements formulated into developer-ready Epics and User Acceptance Testing criteria.

  • US 1.1: As an Admin, I want to create user accounts and assign branch-specific roles so that I can control data access and operational scoping.
  • US 1.2: As a User, I want a secure login and an automated "Forgot Password" link so that I can manage my access without frequent IT intervention.
  • Delegation Rule: If a Manager toggles a Rep to "On Leave", a mandatory prompt asks the Manager to "Select a representative for temporary delegation" to handle the absent rep's active pipeline.
  • US 2.1: As a User, I want to create a Lead with mandatory fields (Name, Phone, Service, Source) so that we maintain high data quality.
  • US 2.2: As a Manager, I want the system to use Round Robin distribution for admin-added leads while respecting "On Leave" statuses.
  • Phase 2 WhatsApp View: "WhatsApp View" Tab embedded inside the Deal/Lead profile to read synced conversation history (Read-only iframe/API) tied to client phone number.
  • US 3.1: As a Sales Rep, I want to move deals through visual stages on a Kanban board.
  • US 3.2: As the System, when a deal is moved to "Qualified," I want to trigger a conversion process that creates a Facility and Client record.
  • Quotation Engine: Auto-Pricing Excel Catalog integration and Dynamic Currency API for accurate order lines.
  • UAT 1 (Discount Workflow): Given a rep applies a 20% discount. When they click Save. Then the system status changes to "Pending Approval", routes to Manager. If Rejected, manager must write notes (reason).
  • UAT 2 (Duplicate Prevention): Given a rep is qualifying a lead. When they enter a CR Number that already exists in the DB. Then system auto-links new Deal to existing Facility.
  • US 4.1: As a Sales Rep, I want to log Calls, Emails, and Meetings on a deal’s profile.
  • US 4.2: As a Manager, I want the system to flag any deal with 96 hours (4 Absolute Days) of inactivity so I can intervene.
  • UAT 4 (SLA Alert): Given a deal has no activity for 96 hours. When system checks inactivity, Then: Alert is sent, Deal marked as warning.
  • US 5.1: As an After-Sales Rep, I want automated alerts for certificates expiring in 30 days or 7 months.
  • US 5.2: As a Manager, I want a weekly report on pending renewals.
  • UAT 5 (Renewal Flow): Given a certificate is expiring. When it reaches threshold, Then: Alert triggered. One-Click Renewal deal can be created prepopulating Facility/Client data.
  • Audit Log Viewer (سجل التعديلات): "As an Admin, I want a 'History Log' button on every Deal to see who changed what and when, so that disputes over deal ownership or manual edits are easily resolved."
  • Pipeline Search & Filters: "As a User, I want a sticky search bar to find deals instantly by CR Number, Company Name, or Phone, so that navigation is seamless."
